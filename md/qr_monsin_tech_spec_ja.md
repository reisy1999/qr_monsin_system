# 📄 技術仕様書：QR問診搬送システム

## 🧭 システム概要

患者がスマートフォンで問診フォームに回答し、データをローカル処理でQRコード化し、来院時に院内専用端末でそのQRを読み取って復元・利用する方式。

本システムは以下の2アプリケーションで構成される：

1.  **入力アプリ（外部）**：問診フォームを提供し、ローカル処理でQRを生成。**QR生成完了の事実のみを匿名でサーバーに通知する。**
2.  **復元アプリ（院内）**：QRコードを専用端末で読み取り、問診データを復元・表示。

---

## 1. 🔧 アプリケーション構成

### 1.1 入力アプリ（スマートフォン対応）

| 項目 | 内容 |
| :--- | :--- |
| **提供形式** | **動的サーバー (Node.js/Express等) による配信。** HTML+JS（SPA構成）自体は変更なし。 |
| 入力内容 | 選択式問診（ラジオ、チェックボックス、プルダウン）、自由記述（目安：最大300文字） |
| **データ処理** | CSV化 → Shift_JIS変換 → zlib圧縮 → **公開鍵による暗号化 (RSA-OAEP)** → Base64エンコード → QRコード生成 |
| QR仕様 | ECCレベル M, Version 自動（最大 2953 バイト） |
| **セキュリティ** | **公開鍵暗号方式を採用。サーバーから配布される公開鍵で暗号化を行う。暗号化に使われる公開鍵が漏洩しても、秘密鍵がなければ復号はできない。** |
| 状態管理 | Cookie, LocalStorage, SessionStorage 一切使用禁止 |
| **通信** | **テンプレートと公開鍵の配信、及びQR生成完了イベントのログ送信以外の通信は行わない。** |
| **API通信** | **QRコード生成完了時、サーバーのログ記録用APIエンドポイントにPOSTリクエストを送信する。（詳細は8. API仕様を参照）** |
| 対応端末 | iOS Safari / Android Chrome |

### 1.2 復元アプリ（イントラネット端末）

| 項目 | 内容 |
| :--- | :--- |
| **動作場所** | **院内閉域ネットワーク上の動的サーバーで動作。**（インターネット非接続） |
| 入力方法 | 専用QRスキャナ（USB/HID）からの文字列入力 |
| **データ復元** | QR読み取り → Base64復号 → **秘密鍵による復号** → zlib解凍 → Shift_JISからUTF-8へ変換 → CSV整形 |
| データ表示 | UI上に復元データを表示。**ローカル端末への保存は不可とする。** |
| **セキュリティ** | **復号には、院内サーバーに厳重に保管された秘密鍵を使用する。秘密鍵は院外に一切持ち出さない。** |

## 2. 📦 データ構造

### 2.1 QRに埋め込まれるCSV構造（例）

```
サトウタロウ,1979,03,25,30,1,3;4;6,2,8;1;2,今すぐ受診させてほしい
```

※ 文字列（氏名）以外はすべて数値化。自由記述欄は目安として最大300文字程度まで。セミコロン区切りで複数選択可。選択式問診を中心とし、自由記述欄の使用は必要最小限に抑えることが望ましい。

### 2.2 圧縮・暗号化

| ステップ | 処理 |
| :--- | :--- |
| 1 | CSV文字列（UTF-8）を生成 |
| 2 | Shift_JISへエンコード |
| 3 | zlib圧縮 |
| 4 | **公開鍵で暗号化** |
| 5 | Base64エンコード → QR生成 |

## 3. 🔐 セキュリティ方針

* 問診データ本体は**患者端末から一切送信されません。**
* 問診データ本体は、QRコードによる物理的な持参を唯一の搬送手段とします。**QRコード生成が完了したという匿名の事実のみを、統計目的でサーバーに通知します。**
* 外部APIの導入は現状想定されておらず、仮に将来導入される場合でも**個人情報や問診本文データを外部に送信することは一切ありません。**

---

## 4. ⚠️ 設計上の注意点（全体）

1.  選択式主体：データサイズ削減と誤記防止（数値ID形式で構造化）
2.  自動バリデーション：未入力、矛盾ロジックを**ローカル**で事前検出
3.  全角・半角変換：数値・カタカナはすべて半角/全角正規化（NFKC）
4.  自由記述欄の文字列に読み取り不能な文字が含まれていた場合の動作を定義し、可能であればローカルで排除・警告できるようにする
5.  QR読み取り不能対策：ECCレベルM、白黒コントラスト、画面スクショ誘導
6.  攻撃モデル：患者のQRが流出し、かつそのQRが本アプリで作成されたことが判明している場合、復号可能であることを前提としたリスク評価が必要
7.  **鍵管理：秘密鍵は院内サーバーから絶対に漏洩させてはならない。キーペアは毎年6月1日に更新する。6月1日から7月1日までの移行期間中は、復元アプリは新旧両方の秘密鍵を保持し、QRデータの復号を試行すること。7月2日以降は新しい秘密鍵のみを使用する。**
8.  空白のまま処理されたときにどう返すかを設計しておくこと

## 5. 🌐 ネットワーク分離ポリシー

開発環境では、`input-app`と`restore-app`は別々のポート（例: `http://localhost:5173`, `http://localhost:5174`）で実行されることがあります。

本番環境では、`input-app`は**外部公開サーバー**に、`restore-app`は**閉域イントラネット**内にそれぞれ配置されることを前提とします。

## 6. 💾 データ永続性とロギング

### 6.1 クライアントサイド

アプリケーションは `sessionStorage`、`localStorage`、`IndexedDB` を含むがこれに限定されない、いかなる形式のクライアントサイドストレージも使用しません。

### 6.2 サーバーサイド

セキュリティ監査と利用状況統計を目的として、サーバーサイドでログを記録します。

* **アクセスログ**: 全てのリクエストに対して、IPアドレス、タイムスタンプ、リクエストパス、ユーザーエージェント等を記録します。
* **アプリケーションログ**: QR生成完了通知API(`POST /api/logs/qr-generated`)へのリクエスト内容を記録します。
* **ログ非含有情報**: いずれのログにも、**問診内容、暗号化データ、Base64文字列、暗号鍵そのもの**を一切含まないことを保証します。

## 8. 📊 API仕様 (ログ記録用)

本システムは利用状況の統計を目的として、以下のAPIエンドポイントを設置します。

### 8.1 QR生成完了ログ

QRコードが正常に生成されたことを記録するためのエンドポイントです。

* **エンドポイント**: `POST /api/logs/qr-generated`
* **送信タイミング**: QRコードがブラウザ上に表示された直後
* **リクエストボディ (Content-Type: application/json)**:
    ```json
    {
      "formVersion": "string",
      "timestamp": "string"
    }
    ```
* **フィールド説明**:
    * `formVersion` (string): フォームのバージョンを示す文字列。 (例: `"1.3.0"`)
    * `timestamp` (string): QR生成が完了した時点のタイムスタンプ (ISO 8601形式)。 (例: `"2025-07-01T15:24:00+09:00"`)
* **注意**: このAPI通信には、個人を特定しうる情報（氏名、UUID等）や、フィンガープリントに繋がりうる情報（`userAgent`等）、問診の入力内容を一切含めてはなりません。

## 9. 📝 エラーハンドリング方針

予見されるエラーに対し、アプリケーションは以下の方針で一貫した振る舞いをすること。

### 9.1 入力アプリ（患者側）

| エラー状況 | 表示メッセージ（例） | 備考 |
| :--- | :--- | :--- |
| **公開鍵の取得失敗** | 「サーバーとの通信に失敗しました。通信環境の良い場所でページを再読み込みしてください。」 | ネットワーク起因のエラー。ユーザーに再試行を促す。 |
| **QRコード生成失敗** | 「QRコードの生成に失敗しました。お手数ですが、最初からやり直してください。」 | データサイズ超過やライブラリ内部エラーなど。 |

### 9.2 復元アプリ（院内側）

| エラー状況 | 表示メッセージ（例） | 備考 |
| :--- | :--- | :--- |
| **QRデータ破損・不正** | 「このQRコードは読み取れません。QRコードが欠けていないか、有効なものか確認してください。」 | Base64デコード、zlib解凍、復号処理のいずれかで失敗した場合。 |
| **復号失敗（鍵不一致）** | 「復号に失敗しました。有効期間外のQRコードである可能性があります。患者様にご確認ください。」 | 保持している全ての秘密鍵で復号を試行し、失敗した場合。 |

## 10. 💻 技術スタック（使用ライブラリ）

本システムの実装には、以下のライブラリおよび技術を使用することを規範とする。

| 機能 | 技術・ライブラリ | 対象アプリ | 選定理由 |
| :--- | :--- | :--- | :--- |
| **公開鍵暗号** | Web Crypto API | 入力/復元 | ブラウザ・Node.jsの標準APIであり、追加ライブラリ不要で高いセキュリティを確保できる。 |
| **zlib圧縮** | pako.js | 入力 | 軽量でブラウザでの実績が豊富。 |
| **zlib解凍** | zlib (Node.js標準) | 復元 | Node.jsの標準モジュールであり、安定性とパフォーマンスに優れる。 |
| **QRコード生成** | qrcode | 入力 | 広く採用されており、React環境との親和性も高い。 |
| **文字コード変換** | encoding-japanese | 入力 | ブラウザでのShift_JIS変換において信頼性が高い。 |
| **文字コード変換** | iconv-lite | 復元 | Node.jsでの文字コード変換の定番ライブラリ。 |
| **サーバーログ** | pino / winston | 入力/復元 | Node.jsでの構造化ロギングの標準的な選択肢。 |
| **環境変数管理** | dotenv | 入力/復元 | 開発環境での機密情報管理を簡素化する。 |
