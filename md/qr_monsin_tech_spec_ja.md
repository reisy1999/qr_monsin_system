# 📄 技術仕様書：QR問診搬送システム

## 🧭 システム概要
患者がスマートフォンで問診フォームに回答し、データをローカル処理でQRコード化し、来院時に院内専用端末でそのQRを読み取って復元・利用する方式。

本システムは以下の2アプリケーションで構成される：

1.  **入力アプリ（外部）**：問診フォームを提供し、ローカル処理でQRを生成
2.  **復元アプリ（院内）**：QRコードを専用端末で読み取り、問診データを復元・表示

---

## 1. 🔧 アプリケーション構成

### 1.1 入力アプリ（スマートフォン対応）
| 項目 | 内容 |
|---|---|
| 提供形式 | HTML+JS（SPA構成）またはPWA（将来対応） |
| 入力内容 | 選択式問診（ラジオ、チェックボックス、プルダウン）、自由記述（最大300文字まで） |
| データ処理 | CSV化 → Shift_JIS変換 → zlib圧縮 → AES暗号化 → Base64エンコード → QRコード生成 |
| QR仕様 | ECCレベル M, Version 自動（最大 2953 バイト） |
| セキュリティ | AES-256-CBC 暗号化（復号鍵はイントラにのみ保持） |
| 状態管理 | Cookie, LocalStorage, SessionStorage 一切使用禁止（完全揮発性） |
| 通信 | 原則非通信。将来的に「ゼロ知識証明API（署名 or ハッシュのみ）」を導入する可能性あり |
| 対応端末 | iOS Safari / Android Chrome |

### 1.2 復元アプリ（イントラネット端末）
| 項目 | 内容 |
|---|---|
| 動作場所 | 院内閉域ネットワーク上（インターネット非接続） |
| 入力方法 | 専用QRスキャナ（USB/HID）からの文字列入力 |
| データ復元 | QR読み取り → Base64復号 → AES復号 → zlib解凍 → Shift_JISからUTF-8へ変換 → CSV整形 |
| データ表示 | 復元されたデータは.txtファイルとしてフォーマットされ、ダウンロードされます。データはUIに表示されません。 |
| セキュリティ | 復号鍵はローカル固定。アプリは.txtファイル生成後、すべての復号済みデータを即座に破棄します。 |

---

## 2. 📦 データ構造

### 2.1 QRに埋め込まれるCSV構造（例）
```
サトウタロウ,1979,03,25,30,1,3;4;6,2,8;1;2,今すぐ受診させてほしい
```
※ 文字列（氏名）以外はすべて数値化。自由記述欄は最大300文字。セミコロン区切りで複数選択可。

### 2.2 圧縮・暗号化
| ステップ | 処理 |
|---|---|
| 1 | CSV文字列（UTF-8）を生成 |
| 2 | Shift_JISへエンコード |
| 3 | zlib圧縮（pako.js） |
| 4 | AES暗号化（CryptoJSなど） |
| 5 | Base64エンコード → QR生成 |

## 3. 🔐 セキュリティ方針

-   問診データ本体は**患者端末から送信されない**
-   外部APIを使用する場合も**個人情報や本文データは一切送信しない**
-   送信可能なのは以下のような構造化メタ情報：
    ```json
    {
      "hash": "xxxxxxxx",
      "valid": true,
      "id": "12345678",
      "version": "1.0"
    }
    ```
    ```
-   患者端末は「全処理完了後に画面遷移・離脱」でデータ破棄。再利用防止
-   HMAC検証は利用しない

---

## 4. ⚠️ 設計上の注意点（全体）

1.  選択式主体：データサイズ削減と誤記防止（数値ID形式で構造化）
2.  自動バリデーション：未入力、矛盾ロジックをローカルで事前検出
3.  全角・半角変換：数値・カタカナはすべて半角/全角正規化（NFKC）
4.  自由記述欄制限：文字数上限（300文字） + 攻撃耐性チェック（emoji除外など）
5.  QR読み取り不能対策：ECCレベルM、白黒コントラスト、画面スクショ誘導
6.  攻撃モデル：最悪でも流出するのは暗号化問診と署名済みハッシュのみ
7.  鍵管理：どちらもAESであるため、そもそも復号される前提で進めること
8.  空白のまま処理されたときにどう返すかを設計しておくこと

## 6. 🌐 ネットワーク分離ポリシー

開発環境では、`input-app`と`restore-app`は別々のポート（例: http://localhost:5173, http://localhost:5174）で実行されることがあります。

しかし、本番環境では以下の通りとします。

-   `input-app`は**公開された外部サーバー**（例: https://monsin.example.com）にデプロイする必要があります。
-   `restore-app`は**病院の内部閉域ネットワーク（イントラネット）内**にのみデプロイし、インターネットから完全に分離する必要があります。

実際の運用環境では、これら2つのアプリケーション間で**直接的または間接的な通信が一切あってはなりません**。

このネットワーク分離は、患者のプライバシーを維持し、データの完全性を確保し、共有インフラを介した漏洩や注入を防ぐために不可欠です。

---

## 8. データ永続性

-   **ストレージ不使用:** アプリケーションは`sessionStorage`、`localStorage`、`IndexedDB`を含むがこれに限定されない、いかなる形式のストレージも使用してはなりません。
-   **ロギング・キャッシュ不使用:** いかなるロギングやファイルベースのキャッシュも許可されません。
