# 📄 技術仕様書：QR問診搬送システム



## 🧭 システム概要



患者がスマートフォンで問診フォームに回答し、データをローカル処理でQRコード化し、来院時に院内専用端末でそのQRを読み取って復元・利用する方式。



本システムは以下の2アプリケーションで構成される：



1. **入力アプリ（外部）**：問診フォームを提供し、ローカル処理でQRを生成

2. **復元アプリ（院内）**：QRコードを専用端末で読み取り、問診データを復元・表示



---



## 1. 🔧 アプリケーション構成



### 1.1 入力アプリ（スマートフォン対応）



| 項目     | 内容                                                                                                        |

| ------ | --------------------------------------------------------------------------------------------------------- |

| 提供形式   | HTML+JS（SPA構成）                                                                                            |

| 入力内容   | 選択式問診（ラジオ、チェックボックス、プルダウン）、自由記述（目安：最大300文字）                                                                |

| データ処理  | CSV化 → Shift\_JIS変換 → zlib圧縮 → AES暗号化（CryptoJS.AES.encrypt を使用） → Base64エンコード → QRコード生成                   |

| QR仕様   | ECCレベル M, Version 自動（最大 2953 バイト）                                                                         |

| セキュリティ | CryptoJS.AES.encrypt による暗号化を採用。これは暗黙的にIV（初期化ベクトル）を生成し内部で処理される形式である。|

| 状態管理   | Cookie, LocalStorage, SessionStorage 一切使用禁止                                                        |

| 通信     | テンプレートと暗号鍵配信以外のインターネット通信は一切行わない                                                                           |

| 対応端末   | iOS Safari / Android Chrome                                                                               |



### 1.2 復元アプリ（イントラネット端末）



| 項目     | 内容                                                                                                           |

| ------ | ------------------------------------------------------------------------------------------------------------ |

| 動作場所   | 院内閉域ネットワーク上（インターネット非接続）                                                                                      |

| 入力方法   | 専用QRスキャナ（USB/HID）からの文字列入力                                                                                    |

| データ復元  | QR読み取り → Base64復号 → AES復号 → zlib解凍 → Shift\_JISからUTF-8へ変換 → CSV整形                                            |

| データ表示  | 現在はUI上に復元データをテーブル表示し、コピー・ダウンロード機能を提供していますが、個人情報保護の観点から**今後はUI表示のみを維持し、ローカル端末への保存不可とするような実装への移行を検討しています**。 |

| セキュリティ | 復号鍵は暗号鍵と**完全に同一**。                                                     |

## 2. 📦 データ構造



### 2.1 QRに埋め込まれるCSV構造（例）



```

サトウタロウ,1979,03,25,30,1,3;4;6,2,8;1;2,今すぐ受診させてほしい

```



※ 文字列（氏名）以外はすべて数値化。自由記述欄は目安として最大300文字程度まで。セミコロン区切りで複数選択可。選択式問診を中心とし、自由記述欄の使用は必要最小限に抑えることが望ましい。



### 2.2 圧縮・暗号化



| ステップ | 処理                           |

| ---- | ---------------------------- |

| 1    | CSV文字列（UTF-8）を生成             |

| 2    | Shift\_JISへエンコード             |

| 3    | zlib圧縮（pako.js）              |

| 4    | AES暗号化（CryptoJS.AES.encrypt） |

| 5    | Base64エンコード → QR生成           |

## 3. 🔐 セキュリティ方針



* 問診データ本体は**患者端末から一切送信されません**

* 本システムは「QRコードによる物理的な持参」を唯一の搬送手段とし、**インターネット経由での提出・アップロード・送信を完全に禁止**しています

* 外部APIの導入は現状想定されておらず、仮に将来導入される場合でも**個人情報や問診本文データを外部に送信することは一切ありません**

## 4. ⚠️ 設計上の注意点（全体）



1. 選択式主体：データサイズ削減と誤記防止（数値ID形式で構造化）

2. 自動バリデーション：未入力、矛盾ロジックを**ローカル**で事前検出

3. 全角・半角変換：数値・カタカナはすべて半角/全角正規化（NFKC）

4. 自由記述欄の文字列に読み取り不能な文字が含まれていた場合の動作を定義し、可能であればローカルで排除・警告できるようにする

5. QR読み取り不能対策：ECCレベルM、白黒コントラスト、画面スクショ誘導

6. 攻撃モデル：患者のQRが流出し、かつそのQRが本アプリで作成されたことが判明している場合、復号可能であることを前提としたリスク評価が必要

7. 鍵管理：どちらもAESであるため、そもそも復号される前提で進めること

8. 空白のまま処理されたときにどう返すかを設計しておくこと



---



## 5. 🌐 ネットワーク分離ポリシー



開発環境では、`input-app`と`restore-app`は別々のポート（例: [http://localhost:5173](http://localhost:5173), [http://localhost:5174）で実行されることがあります。](http://localhost:5174）で実行されることがあります。)



本番環境では、`input-app`は**外部公開サーバー**に、`restore-app`は**閉域イントラネット**内にそれぞれ配置されることを前提とします。



---



## 6. データ永続性



* **ストレージ不使用:** アプリケーションは`sessionStorage`、`localStorage`、`IndexedDB`を含むがこれに限定されない、いかなる形式のストレージも使用しないことが設計思想です。

* **ロギング・キャッシュ不使用:** いかなるロギングやファイルベースのキャッシュも許可されません。
